# 2. Verktyg och teknologier

### 2.1 Backend – ASP.NET Core Web API

Backend-delen av systemet är byggd med **ASP.NET Core Web API**. API:et ansvarar för all affärslogik, datavalidering, autentisering och kommunikation med databasen. Genom att separera backend från frontend uppnås en tydlig ansvarsfördelning samt möjlighet att vidareutveckla systemet med andra klienter i framtiden.

API:et exponerar REST-baserade endpoints för bland annat:

* användarregistrering och inloggning
* hantering av BRF:er
* formulärinlämningar
* köpflöden
* administrativ granskning av avvikelser

Denna arkitektur möjliggör testning via verktyg som Swagger och förenklar felsökning och vidareutveckling.

### 2.2 Entity Framework Core och datamodellering

För databaskommunikation används **Entity Framework Core (EF Core)** som Object-Relational Mapper (ORM). EF Core möjliggör arbete med databasen via C#-klasser istället för rå SQL, vilket ger:

* stark typkontroll
* bättre läsbarhet
* minskad risk för SQL-injektion
* enklare underhåll

Datamodellen är uppbyggd kring centrala entiteter såsom:

* User
* Brf
* FormSubmission
* Purchase
* BrfAvvikelse

Relationer mellan entiteter används för att säkerställa spårbarhet, exempelvis vilken mäklare som köpt vilken information eller vilket formulär som hör till en viss BRF.

### 2.3 Databaser – SQLite och PostgreSQL

Projektet använder olika databaser beroende på miljö:

#### Lokal utveckling

Under lokal utveckling används **SQLite**, vilket ger en lättviktig och filbaserad databas som är enkel att sätta upp och snabb att arbeta med under utvecklingsfasen.

#### Produktion

I produktionsmiljö används **PostgreSQL**, en robust relationsdatabas som lämpar sig för fleranvändarsystem, transaktioner och framtida skalning. PostgreSQL används tillsammans med automatiska databasmigreringar för att säkerställa att databasens struktur alltid är i synk med applikationens datamodell.

Denna uppdelning mellan lokal och produktionsdatabas gör utvecklingsprocessen smidig utan att kompromissa med stabilitet i drift.

### 2.4 Frontend – Blazor Server

Frontend är byggd med **Blazor Server**, vilket innebär att användargränssnittet skrivs i C# och Razor istället för JavaScript-baserade ramverk. Blazor Server valdes för att:

* möjliggöra återanvändning av C#-logik mellan frontend och backend
* minska behovet av JavaScript
* förenkla state-hantering
* erbjuda server-side rendering med god prestanda

Blazor-komponenter används för formulär, inloggning, navigation och vyer för olika användarroller.

### 2.5 UI-ramverk – MudBlazor

För användargränssnittet används **MudBlazor**, ett komponentbibliotek för Blazor som tillhandahåller färdiga UI-komponenter med konsekvent design och tillgänglighet i fokus.

MudBlazor används bland annat för:

* formulärfält
* knappar
* kort (cards)
* layout och responsiv design

Detta har möjliggjort ett professionellt utseende utan att behöva bygga alla UI-komponenter från grunden.

### 2.6 Säkerhet – autentisering och lösenordshantering

Säkerhet är en central del av systemets design. Följande säkerhetsåtgärder har implementerats:

* **BCrypt** används för hashning av lösenord
* Lösenord lagras aldrig i klartext
* Roller kontrolleras på backend
* Endast behöriga användare får tillgång till skyddade vyer
* Administrativa funktioner är inte exponerade i frontend

BCrypt valdes eftersom det är en beprövad och säker standard för lösenordshantering, med inbyggt skydd mot brute-force-attacker.

***

### 2.7 Infrastruktur och deployment

För deployment och drift används containerbaserad teknik med **Docker**. Applikationen körs i separata containers för backend och frontend, vilket ger:

* konsekvent körmiljö
* enklare felsökning
* möjlighet till framtida skalning

I produktionsmiljö används en molnbaserad plattform som stödjer Docker, automatiska byggen och HTTPS, vilket möjliggör kontinuerlig integration och deployment (CI/CD).

### 2.8 Versionshantering

All kod versionshanteras med **Git** och lagras i ett GitHub-repository. Git används för:

* spårbarhet av ändringar
* säker rollback vid fel
* parallellt arbete i olika brancher

Detta har varit särskilt viktigt under utvecklingsfasen då flera större tekniska förändringar genomförts, exempelvis migrering av databas och förändringar i deploymentstrategi.
