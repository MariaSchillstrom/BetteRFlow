# 8. Framtida utveckling

### 8.1 Resultat

Projektet har resulterat i en fungerande webbaserad applikation som uppfyller de uppsatta målen. BetteRFlow digitaliserar informationsflödet mellan bostadsrättsföreningar och mäklare och ersätter manuella processer med ett strukturerat och spårbart system.

De huvudsakliga resultaten är:

* ett fungerande registrerings- och inloggningssystem med rollhantering
* digitala formulär för insamling av BRF-information
* automatisk identifiering av avvikelser mellan grunddata och inrapporterade uppgifter
* ett administrativt flöde för manuell granskning och beslut
* ett förenklat köpflöde för mäklare
* en tydlig separation mellan frontend, backend och datalager

Systemet är driftsatt i molnmiljö och kan köras både lokalt och i produktion.

***

### 8.2 Uppfyllelse av mål

De mål som definierades i projektets inledande fas har i huvudsak uppfyllts:

* Manuell hantering av PDF-formulär har ersatts av digitala formulär
* Informationen lagras strukturerat i databas
* Avvikelsehantering säkerställer datakvalitet
* Roller och behörigheter hanteras säkert
* Lösningen är anpassad för molndrift

Vissa delar, såsom fullständig betalningsintegration och avancerad statistik, har medvetet avgränsats för att hålla projektets omfattning rimlig.

### 8.3 Reflektion över utvecklingsprocessen

Utvecklingsarbetet har präglats av ett iterativt arbetssätt där funktionalitet har byggts, testats och förbättrats successivt. En viktig lärdom har varit vikten av tydlig struktur och separation mellan olika delar av systemet.

Arbetet har också inneburit praktiska utmaningar kring:

* konfiguration av olika miljöer (lokalt vs produktion)
* autentisering och rollhantering
* hantering av databasadministration utanför applikationen

Genom att lösa dessa problem har en djupare förståelse för både applikationsutveckling och systemdrift uppnåtts.

### 8.4 Lärdomar

Projektet har bidragit till flera viktiga insikter:

* Vikten av säkerhet redan från start, särskilt kring lösenord och roller
* Fördelarna med att separera frontend och backend
* Betydelsen av tydliga datamodeller och relationer
* Att systemadministration och applikationslogik är två skilda ansvarsområden
* Att dokumentation i tidiga faser underlättar slutrapportering

#### **8.4.1 Vad hade jag gjort annorlunda?**

Med erfarenheten från detta projekt skulle jag prioritera följande annorlunda:

**1. Databas från början**

* ❌ Började med SQLite → migrerade till PostgreSQL
* ✅ Bättre: Använt PostgreSQL från dag 1 (även lokalt med Docker)
* **Varför:** Undvika miljöskillnader och migrations-problem

**2. Migrations-strategi tidigt**

* ❌ Upptäckte sent att migrations inte körde automatiskt på Render
* ✅ Bättre: Researcha deployment-process innan val av plattform
* **Varför:** Manuell databas-hantering är riskabelt i produktion

**3. State management med versionshantering**

* ❌ Ändrade localStorage-struktur → gamla users såg inte menyer
* ✅ Bättre: Version-field i localStorage från start
* **Varför:** Migration paths för befintliga användare måste planeras

**4. Library-first approach för komplex logik**

* ❌ Försökte implementera fuzzy matching från scratch
* ✅ Bättre: Utvärdera färdiga NuGet-packages (FuzzySharp) först
* **Varför:** Återuppfinn inte hjulet - använd beprövade lösningar

**5. Automated testing tidigt**

* ❌ Mestadels manuell testning
* ✅ Bättre: Unit tests för affärslogik från början
* **Varför:** Sparar tid vid refactoring och ändringar

**6. JWT och Identity Framework från start**

* ❌ Förenklad auth med localStorage
* ✅ Bättre: ASP.NET Identity implementation från dag 1
* **Varför:** Svårt att migrera auth-system i efterhand

Dessa lärdomar är direkt överförbara till framtida projekt inom system- och mjukvaruutveckling.

### 8.5 Framtida utveckling



_Idéer för långsiktig utveckling (v 3.5-3.0+) har diskuterats med ChatGPT för_\
_att utforska möjliga tekniska lösningar och arkitektoniska alternativ._

#### Kort sikt (v2.0)

**Migration till Azure SQL med CI/CD**

* Automatiska migrations vid deployment (eliminera manuell pgAdmin-hantering)
* Staging-miljö för testning före produktion
* Azure DevOps eller GitHub Actions pipeline

**Fuzzy matching för organisationsnummer**

* Matcha även vid stavfel i org.nr (Levenshtein distance)
* Multi-field matching (org.nr + namn + adress)
* Färdigt NuGet-package (FuzzySharp)

**Mäklarhantering (samma struktur som BRF)**

* MaklarFirma-tabell för grunddata
* Matchning vid registrering
* Excel-import för bulkhantering
* Admin aktiverar nya mäklare

#### Medellång sikt (v2.5-3.0) -

**Swish-integration**

* Swish Handel API för betalning
* QR-kod och webhook för bekräftelse

**Email-hantering från Admin Dashboard**

* Skicka inbjudningar direkt från systemet (SendGrid/SMTP)
* Tracking: Skickad → Öppnad → Registrerad
* Unika registreringslänkar

**Förbättrad autentisering**

* ASP.NET Identity istället för förenklad auth
* JWT med refresh tokens
* HttpOnly cookies istället för localStorage

**Statistik och dashboards**

* Registreringar över tid
* Konverteringsgrad (inbjudningar → registreringar)
* Populäraste sökningar
* Grafer med Chart.js
* PowerBI

**Automatiserade tester**

* Unit tests (xUnit) för affärslogik
* Integration tests för API
* CI/CD-integration

#### Lång sikt (v3.0+)

**Blazor WebAssembly migration**

* Publika delar (mäklarsökning) som WASM för skalbarhet (10,000+ samtidiga användare)
* Admin-delen behåller Server (färre användare, snabbare development)

**Integration med externa register**

* Bolagsverket API för automatisk hämtning av BRF-data
* Ersätter manuell Excel-import med automatisk synkronisering
* Periodisk uppdatering av grunddata (organisationsnummer, namn, adress)
* Validering av org.nr vid registrering
* **Motivering:** Excel-import är en tillfällig lösning - API-integration ger alltid aktuell data utan manuell hantering

**Formulär-versionshantering**

* Schema evolution-strategi för att hantera ändringar i formulär över tid
* Backward compatibility vid fältändringar

**Skalbarhet & Säkerhet (vid kraftig tillväxt)**

* Microservices-arkitektur
* Redis cache för prestanda
* OWASP audit och penetration testing
* Rate limiting på API

### 8.6 Slutsats

BetteRFlow visar hur ett digitalt system kan effektivisera informationsflöden i fastighetsrelaterade processer genom strukturerad datahantering, tydliga roller och säker arkitektur. Projektet demonstrerar både teknisk kompetens och förståelse för verkliga verksamhetsbehov.

Genom att kombinera moderna webbramverk, säker autentisering och molnbaserad drift har en stabil grund skapats för vidare utveckling och framtida användning.
